# Тест-кейсы

## Позитивный сценарий

**ID:** API-001

**Название:** Создание объявления с валидными значениями

**Предусловие:** \-

**Шаги выполнения:**
1. Выполнить POST запрос к эндпоинту /api/1/item
Передать в теле запроса тестовые данные
Передать в заголовках:
   - Content-Type: application/json
   - Accept: application/json

**Ожидаемый результат:**
- Код ответа 200 OK
- В теле ответа содержится JSON с данными о статусе
- Поле status содержит сообщение "Сохранили объявление - ***", где \*\*\* - это id созданного объявления

**Тестовые данные:**
``` 
{
  "sellerID": 864678,
  "name": "Телефон",
  "price": 90850,
  "statistics": {
    "likes": 100,
    "viewCount": 200,
    "contacts": 300
  }
}
```

**Постусловия:**
- Объявление создано в системе

**Техника тест-дизайна:** Счастливый путь



## Негативный сценарий

**ID:** API-002

**Название:** Создание объявления без обязательного поля sellerID

**Предусловие:** \-

**Шаги выполнения:**
1. Выполнить POST запрос к эндпоинту /api/1/item
Передать в теле запроса тестовые данные, где отсутствует обязательное поле sellerID
Передать в заголовках:
   - Content-Type: application/json
   - Accept: application/json

**Ожидаемый результат:**
- Код ответа 400 Bad Request
- В теле ответа содержится JSON с сообщением об ошибке
- Поле result.message содержит текст "поле sellerID обязательно"

**Тестовые данные:**
``` 
{
  "name": "Телефон",
  "price": 90850,
  "statistics": {
    "likes": 100,
    "viewCount": 200,
    "contacts": 300
  }
}
```

**Постусловия:**
- Объявление не создано в системе

**Техника тест-дизайна:** Классы эквивалентности



## Корнер-кейс

**ID:** API-003

**Название:** Многократное получение одного и того же объявления

**Предусловие:** 
- Создано объявление через POST /api/1/item
- Получен id созданного объявления

**Шаги выполнения:**
1. Выполнить GET запрос к /api/1/item/{id} - первый раз
2. Выполнить GET запрос к /api/1/item/{id} - второй раз
3. Выполнить GET запрос к /api/1/item/{id} - третий раз

**Ожидаемый результат:**
- Все три запроса возвращают 200 OK
- Тело ответа идентично во всех трёх запросах

**Постусловия:** \-

**Техника тест-дизайна:** Предугадывание ошибок



## Функциональная проверка

**ID:** API-004

**Название:** Полный жизненный цикл объявления (CRUD операция)

**Предусловие:** \-

**Шаги выполнения:**
1. Выполнить POST запрос к /api/1/item. Передать в теле запроса тестовые данные. Сохранить id из ответа
2. Выполнить GET запрос к /api/1/item/{id}. Проверить, что данные соответствуют созданным
3. Выполнить GET запрос к /api/1/{sellerId}/item. Проверить, что созданный id присутствует в списке
4. Выполнить GET запрос к /api/2/statistic/{id}. Проверить, что данные (likes, viewCount, contacts) соответствуют созданным
5. Выполнить DELETE запрос к /api/2/item/{id}

**Ожидаемый результат:**
- Шаг 1: 200 OK, получен id
- Шаг 2: 200 OK, все поля соответствуют созданным
- Шаг 3: 200 OK, созданный id присутствует в массиве
- Шаг 4: 200 OK, ответ — массив, содержит поля likes, viewCount, contacts
- Шаг 5: 200 OK

**Тестовые данные:**
``` 
{
  "sellerID": 864678,
  "name": "Телефон",
  "price": 90850,
  "statistics": {
    "likes": 100,
    "viewCount": 200,
    "contacts": 300
  }
}
```

**Постусловия:** 
- Объявление удалено из системы

**Техника тест-дизайна:** Предугадывание ошибок



## Нефункциональная проверка

**ID:** API-005

**Название:** Время ответа API при создании объявления

**Предусловие:** \-

**Шаги выполнения:**
1. Зафиксировать время начала запроса (start_time)
2. Выполнить POST запрос к эндпоинту /api/1/item. Передать в теле запроса тестовые данные.
3. Зафиксировать время получения ответа (end_time)
4. Вычислить разницу: response_time = end_time - start_time
5. Повторить шаги 1-4 10 раз
6. Вычислить среднее время ответа

**Ожидаемый результат:**
- Код ответа 200 OK для всех запросов
- Среднее время ответа не превышает 500 миллисекунд

**Тестовые данные:**
``` 
{
  "sellerID": 864678,
  "name": "Телефон",
  "price": 90850,
  "statistics": {
    "likes": 100,
    "viewCount": 200,
    "contacts": 300
  }
}
```

**Постусловия:** 
- Созданные объявления остаются в системе



## Проверка безопасности

**ID:** API-006

**Название:** Защита от XSS-инъекции в поле name

**Предусловие:** \-

**Шаги выполнения:**
1. Выполнить POST запрос к /api/1/item с XSS-инъекцией в поле name
2. Сохранить полученный id из ответа
3. Выполнить GET запрос к /api/1/item/{id}
4. Проверить содержимое поля name в ответе

**Ожидаемый результат:**
- Шаг 1: 200 OK, объявление создано
- Шаг 3: 200 OK
- Поле name в ответе содержит экранированную версию скрипта (например, \u003Cscript\u003Ealert('XSS')\u003C/script\u003E) или скрипт удалён

**Тестовые данные:**
``` 
{
  "sellerID": 864678,
  "name": "<script>alert('XSS')</script>",
  "price": 90850,
  "statistics": {
    "likes": 100,
    "viewCount": 200,
    "contacts": 300
  }
}
```

**Постусловия:** 
- Созданные объявления остаются в системе

**Техника тест-дизайна:** Предугадывание ошибок
